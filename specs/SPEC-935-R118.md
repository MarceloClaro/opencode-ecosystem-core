# SPEC-935-R118 — Correção do Handshake MCP em metacognitive-interconnect

## Objetivo

Corrigir a falha real do servidor MCP `metacognitive-interconnect` (`mci/mcp_server.py`): qualquer cliente MCP real (OpenCode CLI, Claude Code, Antigravity) reporta a conexão como falha porque o servidor não implementa o método `initialize`, obrigatório no início de toda sessão MCP.

## Motivação

Reportado pelo usuário: "metacognitive-interconnect failed". Reproduzido enviando um `initialize` real via stdin: o servidor respondia `{"isError": true, "content": [{"type": "text", "text": "Método não suportado"}]}` em vez do handshake esperado — o cliente nunca chega a `tools/list`, então as 4 ferramentas (`mci_register_agent`, `mci_post_task`, `mci_get_memory`, `mci_get_blackboard_state`) nunca ficam disponíveis.

Causa raiz: `SimpleMCPServer.handle_request()` só tratava `tools/list` e `tools/call`, caindo no fallback genérico de erro para qualquer outro método, incluindo `initialize`. Adicionalmente, o loop `run_stdio()` sempre enviava uma resposta JSON-RPC mesmo para notificações (mensagens sem `id`, como `notifications/initialized`), o que viola a especificação JSON-RPC/MCP e pode confundir clientes estritos.

## Escopo

### Em escopo
- `mci/mcp_server.py::SimpleMCPServer.handle_request()`: implementar `initialize` (retornando `protocolVersion`/`capabilities`/`serverInfo`) e `ping`
- `mci/mcp_server.py::SimpleMCPServer.run_stdio()`: suprimir resposta para requisições sem `id` (notificações)
- Testes de regressão cobrindo o handshake completo

### Fora de escopo
- `integrations/antigravity/antigravity_mcp_server.py` — já implementa `initialize`/`notifications/initialized` corretamente, não precisa de correção
- Migração para um SDK MCP oficial (`mcp` PyPI package) em vez da implementação `SimpleMCPServer` caseira — funcional após esta correção, migração é um refactor maior fora de escopo deste fix

## Critérios de Aceitação

1. `initialize` retorna `protocolVersion`, `capabilities` e `serverInfo` válidos, nunca `isError: true`
2. O `protocolVersion` solicitado pelo cliente é ecoado de volta (compatibilidade com múltiplas versões do protocolo)
3. `ping` responde sem erro
4. `tools/list` e `tools/call` continuam funcionando normalmente após o handshake
5. Requisições sem `id` (notificações, ex. `notifications/initialized`) não geram nenhuma resposta JSON-RPC no loop `run_stdio()`
6. Requisições com `id` continuam recebendo resposta normalmente (a supressão é específica de notificações, não geral)
7. Suíte de testes existente permanece verde

## Estratégia de Verificação

- `tests/test_r118_mcp_initialize_handshake.py` (6 testes): handshake completo, eco de protocolVersion, ping, tools/list pós-handshake, supressão de resposta a notificação, não-supressão de resposta a requisição com id
- Reprodução manual via stdin: sequência `initialize` → `notifications/initialized` → `tools/list`, confirmando exatamente uma linha de resposta por requisição com `id`
- `python3 -m pytest tests/ -q` completo

## Entrega Esperada

O servidor MCP `metacognitive-interconnect` completa o handshake corretamente com qualquer cliente MCP real, eliminando o erro "failed" relatado.
