# SPEC-935-R125 — MIRA como parte de primeira classe do ecossistema

## Objetivo

Tornar o pipeline MIRA de apresentações (R123) alcançável e visível em
todas as superfícies do ecossistema — não apenas um método solto no
orquestrador. Expor `orchestrator.present()` no CLI (comando direto +
opção de menu), atualizar os textos de ajuda, e documentar o subsistema
no README, no ARCHITECTURE e no mapa 3D, conectando-o aos agent cards
MIRA que **já existem** no catálogo.

## Motivação

Pedido do usuário: "quero mira como parte do ecossistema".

Levantamento do estado atual:
- **R123** criou o motor executável (`illustrations/mira_deck.py::MiraDeckPipeline`)
  e o método `MarceloClaroOrchestrator.present(production_folder)` — mas
  esse método **não tem nenhum ponto de entrada para o usuário**: não há
  comando de CLI nem opção de menu que o invoque.
- O catálogo **já contém** os agent cards do método MIRA (`mira-extract`,
  `mira-planner`, `mira-copywriter`, `mira-builder`, `mira-animator`,
  `mira-validator`, além de `37_agente_apresentacao_slides_banca`) — eles
  são registrados por `register_catalog_agents`. Ou seja, os agentes
  existem, mas o pipeline executável que realiza o método não é
  disparável pelas superfícies do ecossistema.
- Nenhum documento de arquitetura (README/ARCHITECTURE/mapa 3D) menciona
  o subsistema de apresentações MIRA nem o comando para usá-lo.

O elo que falta é de **exposição/ativação**, não de implementação: ligar
o pipeline já pronto às portas de entrada do usuário e à documentação.

## Escopo

### Em escopo
- `marceloclaro/cli.py`:
  - Comando direto `apresentacao` (aliases `present`, `mira`):
    `python3 -m marceloclaro.cli apresentacao <pasta_da_producao>` →
    chama `orchestrator.present(pasta)` e imprime o resultado; sem
    argumento de pasta, imprime uso e sai com código 1.
  - Opção de menu `[10] Apresentação MIRA (manuscrito → deck animado)`
    com prompt guiado pela pasta da produção; pasta vazia cancela.
  - `MENU`, `AJUDA_TEXT` e a dica de "comando desconhecido" atualizados
    para incluir a apresentação MIRA (com linguagem simples: "transforma
    o manuscrito de uma produção num deck de slides animados").
- `README.md`: bullet de feature do subsistema MIRA (R123/R125) e nó no
  diagrama Mermaid (paridade com ARCHITECTURE).
- `ARCHITECTURE.md`: linha(s) na tabela de specs formais (R123, R125) e
  nó do subsistema de apresentações no diagrama.
- `docs/architecture_map.html`: nó/menção do subsistema MIRA e da opção
  `[10]` no nó da CLI; contagens de testes/ciclos/specs atualizadas.
- `tests/test_r125_mira_cli_integration.py`: testes TDD escritos antes.

### Fora de escopo
- Reescrever o `MiraDeckPipeline` (R123) — já implementado e testado.
- Fluxo interativo de supervisão estágio-a-estágio (aprovação nas juntas
  da esteira) — a API por posto já permite; a casca interativa fica para
  ciclo futuro.
- Registrar um agente executor novo no runtime que "encarne" o pipeline
  (os agent cards MIRA já estão no catálogo; a execução real é o
  `present()`). Um agente-executor acoplado ao pipeline é evolução
  natural para ciclo futuro.

## Critérios de Aceitação

1. `python3 -m marceloclaro.cli apresentacao <pasta>` chama
   `orchestrator.present(<pasta>)` e imprime o resultado; os aliases
   `present` e `mira` funcionam igual.
2. Sem a pasta, o comando imprime uso e sai com código 1.
3. A opção de menu `[10]` invoca `orchestrator.present(pasta)` com a
   pasta informada; pasta vazia cancela sem chamar `present`.
4. `MENU` lista `[10]`, e `AJUDA_TEXT` explica a apresentação MIRA em
   linguagem simples e inclui o comando direto na lista.
5. A dica de "comando desconhecido" menciona `apresentacao`.
6. README e ARCHITECTURE mencionam o subsistema MIRA de apresentações e
   têm paridade de nó no diagrama Mermaid.
7. `docs/architecture_map.html` menciona a apresentação MIRA e a opção
   `[10]`; JS embutido continua válido (`node --check`).
8. Suíte de testes completa permanece verde.
