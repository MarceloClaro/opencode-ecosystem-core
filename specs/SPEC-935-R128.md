# SPEC-935-R128 — Encanamento seguro de chaves LLM (OpenAI) via ambiente

## Objetivo

Habilitar o uso de provedores LLM de nuvem (OpenAI e compatíveis) no
ecossistema de forma **segura por construção**: chaves vivem apenas em
variáveis de ambiente / arquivo `.env` protegido, **nunca** versionadas
nem impressas. Corrigir o risco real de `.env` não estar no `.gitignore`,
fornecer um `.env.example` como modelo, e dar visibilidade via `doctor`
de qual provedor LLM está disponível — sem jamais expor o valor da chave.

## Motivação

Pedido do usuário: habilitar a API da OpenAI para o pipeline de pesquisa
("melhorar a gestão de tokens"). A investigação mostrou que o suporte a
OpenAI **já existe** em `research/llm_client.py` (lê `OPENAI_API_KEY`/
`OPENAI_API_BASE`/`OPENAI_MODEL`; `provider="auto"` prioriza Ollama local
e cai para OpenAI). O que faltava era o **encanamento seguro**:

- **Risco real encontrado:** `.env` NÃO está no `.gitignore` — um arquivo
  `.env` com a chave seria commitável por acidente.
- Não há `.env.example` orientando quais variáveis definir.
- O `doctor` não informa se há provedor LLM configurado (o usuário não
  tem como saber se a chave foi reconhecida).

Princípio inegociável: segredo **não** entra no repositório nem em log.
O ecossistema já lê a chave do ambiente; este ciclo só protege e torna
visível esse fluxo — a chave real é sempre definida pelo usuário.

## Escopo

### Em escopo
- `.gitignore`: ignorar `.env` e variações locais (`.env.local`,
  `.env.*.local`), mantendo `.env.example` versionado (negação `!`).
  Ignorar também formatos comuns de segredo (`*.pem`, `*.key`).
- `.env.example` (novo): modelo com as variáveis **vazias/valores
  padrão** — `OPENAI_API_KEY=`, `OPENAI_API_BASE=`, `OPENAI_MODEL=`,
  `OLLAMA_HOST=`, `OLLAMA_MODEL=` — sem nenhum segredo, com comentário
  de que Ollama local é a via de custo zero e OpenAI é fallback de nuvem.
- `marceloclaro/doctor.py`: novo `_check_llm_providers()` — reporta se o
  Ollama está disponível e se `OPENAI_API_KEY` **está definida** (apenas
  o booleano, **nunca** o valor); `pass` se ao menos um provedor está
  disponível, `warn` (nunca `fail`) se nenhum (enriquecimento por LLM é
  opcional). Adicionado a `run_doctor()`.
- `marceloclaro/env_loader.py` (novo): carregador de `.env` sem
  dependências, chamado no início de `cli.main()`. Injeta as variáveis do
  `.env` (protegido) no ambiente **sem sobrescrever** o que já está no
  shell e **sem imprimir valores** — para que o ecossistema reconheça a
  chave sem exportação manual (importante para **acessibilidade**: o
  usuário não precisa digitar/exportar a chave).
- `tests/test_r128_openai_provider.py`: testes TDD escritos antes.

### Fora de escopo
- `opencode.json` e `integrations/opencode_cli.py`: pertencem à frente
  SPEC-108 (modificados/não commitados) — **não tocar** neste ciclo. O
  bloco `openai` do OpenCode CLI é documentado no `.env.example`, mas a
  edição desses arquivos fica para o dono daquela frente.
- Qualquer manipulação/armazenamento da chave real: fora de escopo por
  princípio — a chave é sempre do usuário, no ambiente dele.
- `research/llm_client.py`: já suporta OpenAI; não precisa de mudança.

## Critérios de Aceitação

1. `git check-ignore .env` retorna sucesso (`.env` é ignorado) e
   `.env.example` **não** é ignorado (permanece versionável).
2. `.env.example` existe, nomeia `OPENAI_API_KEY` e **não** contém
   nenhum valor de chave real (nenhuma string começando com `sk-`).
3. `_check_llm_providers()` retorna `pass` quando `OPENAI_API_KEY` está
   definida (mesmo sem Ollama) e `warn` quando nenhum provedor está
   disponível; nunca `fail`.
4. O `detail` do check **nunca** contém o valor da chave — no máximo um
   indicador booleano ("definida"/"ausente").
5. `run_doctor()` inclui o check `llm_providers` na lista de checks.
6. Nenhum segredo é versionado: a suíte e o `git status` não introduzem
   arquivo com chave.
7. Suíte de testes completa permanece verde.

## Estratégia de Verificação

- TDD: `tests/test_r128_openai_provider.py` escrito antes; vermelho →
  verde.
- `git check-ignore` real para `.env`/`.env.example`.
- `monkeypatch` de `OPENAI_API_KEY` e da disponibilidade do Ollama para
  exercitar `pass`/`warn` sem depender do ambiente.
- Asserção explícita de que o `detail` não vaza o valor da chave.
- `python3 -m pytest tests/ -q` completo.
