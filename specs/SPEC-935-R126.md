# SPEC-935-R126 — Agente-executor MIRA de runtime (delegável pelo Blackboard)

## Objetivo

Fechar o acoplamento formal entre o pipeline MIRA de apresentações (R123)
e o runtime multiagente: um **agente-executor** registrado no Blackboard,
com capacidade própria, que realiza o método MIRA quando uma tarefa de
apresentação é **delegada** — fechando o laço completo
`delegate → match (CFP + atenção) → execute → report_completion`, com o
Trust Engine e a Token Economy aprendendo com o resultado.

## Motivação

Pedido do usuário: "prossiga" — dando sequência à evolução natural
explicitada no evo-34 (R125): "o acoplamento formal entre um
agente-executor do runtime e o pipeline continua sendo evolução natural
para ciclo futuro".

Estado após o R125:
- O pipeline executável existe (`MiraDeckPipeline`, R123) e é alcançável
  pelo usuário via CLI (`present()`/`apresentacao`, R125).
- Os agent cards MIRA estão no catálogo — mas são **cartões de
  markdown**, sem executor real; nenhum deles executa o pipeline quando
  uma tarefa é delegada no Blackboard.
- `orchestrator.present()` roda o pipeline **direto**, fora do laço de
  delegação: não posta tarefa, não passa pelo matching por atenção, não
  aciona Trust/Economy. É uma chamada de biblioteca, não um agente do
  runtime.

O elo faltante é um agente de runtime que **encarne** o pipeline e seja
**delegável** — de modo que "gerar uma apresentação" seja uma tarefa de
primeira classe do Blackboard, sujeita ao mesmo governo (confiança,
economia, reflexão) das demais.

## Escopo

### Em escopo
- `illustrations/mira_agent.py` (novo): `MiraPresentationAgent`
  - identidade fixa: `agent_id="mira-presenter"`, nome, descrição,
    `capabilities=["apresentacao", "apresentacao-mira", "mira-deck",
    "slides-animados"]` (a capacidade `apresentacao-mira` é distintiva:
    garante elegibilidade determinística no matching).
  - `register_payload()` → dict pronto para `metabus.publish("agent.register", ...)`.
  - `execute(context)` → lê `production_folder` do contexto, valida a
    presença de `manuscrito.md`, roda `MiraDeckPipeline.run()` gravando
    `apresentacao/deck.html` + `CONFORMIDADE.md`, e retorna um resultado
    padronizado (`ok`, `passed`, `deck`, `conformidade`, `violations`);
    contexto inválido/manuscrito ausente → `{"ok": False, "error": ...}`.
- `illustrations/__init__.py`: exportar `MiraPresentationAgent`.
- `marceloclaro/orchestrator.py`:
  - Registro do agente no Blackboard na inicialização (quando
    `auto_load_agents`), de modo que ele apareça em `list_agents()` e
    `status()`.
  - `register_mira_agent()` idempotente (não duplica o registro).
  - `present_task(production_folder)`: **delega** a apresentação como
    tarefa do Blackboard (`required_capabilities=["apresentacao-mira"]`),
    **executa** pelo agente-executor e **reporta** a conclusão via
    `report_completion` (Trust/Economy aprendem). Retorna o rastro
    completo (`task_id`, `agent_id`, mais o resultado da execução).
- `tests/test_r126_mira_agent_runtime.py`: testes TDD escritos antes.

### Fora de escopo
- Reescrever `orchestrator.present()` — permanece como a via **direta**
  (chamada de biblioteca, sem delegação); `present_task()` é a via
  **delegada**. As duas coexistem por design.
- Execução assíncrona/concorrente de múltiplas tarefas de apresentação —
  o laço síncrono do MetaBus é suficiente para o acoplamento.
- Expor `present_task` no CLI — o comando `apresentacao` (R125) continua
  usando a via direta; um seletor direto/delegado no CLI fica para ciclo
  futuro.

## Critérios de Aceitação

1. `MiraPresentationAgent.execute({"production_folder": <pasta com
   manuscrito.md>})` gera `apresentacao/deck.html` + `CONFORMIDADE.md` e
   retorna `ok=True`, `passed=True`, com o caminho do deck.
2. `execute` com contexto sem `production_folder` ou sem `manuscrito.md`
   retorna `ok=False` e uma mensagem de erro, sem lançar exceção.
3. Após a inicialização do orquestrador (com `auto_load_agents`), o
   agente `mira-presenter` aparece em `list_agents()` com a capacidade
   `apresentacao-mira`.
4. `register_mira_agent()` é idempotente: chamá-lo duas vezes não
   duplica o cartão no registro do Blackboard.
5. `present_task(<pasta>)` retorna `task_id` e `agent_id="mira-presenter"`,
   gera o deck e a tarefa correspondente no Blackboard fica com status
   `completed` (atribuída a `mira-presenter`).
6. Numa produção sem `manuscrito.md`, `present_task` reporta a tarefa
   como falha (`success=False`) e retorna `ok=False`/`error`, sem
   exceção.
7. A capacidade `apresentacao-mira` é exclusiva do `mira-presenter`
   entre os agentes elegíveis (atribuição determinística).
8. Suíte de testes completa permanece verde.

## Estratégia de Verificação
